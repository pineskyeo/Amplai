## 토큰이 뭔데

LLM은 글자 단위로 읽지 않는다. 토큰이라는 단위로 읽는다.

```
영어: "Hello world"     → ["Hello", " world"]       = 2 토큰
한국어: "안녕하세요"      → ["안녕", "하", "세요"]      = 3 토큰
코드: "console.log()"   → ["console", ".", "log", "()"] = 4 토큰
```

왜 글자도 아니고 단어도 아닌 토큰인가?

- 글자 단위: "Hello"를 이해하려면 H, e, l, l, o 5번 처리. 비효율적.
- 단어 단위: 새 단어가 나올 때마다 사전에 추가해야 함. 비현실적.
- 토큰: BPE(Byte Pair Encoding)로 자주 쓰는 조합을 묶는다. 그 중간.

## 왜 비용이 드는가

토큰 1개를 처리할 때마다 GPU가 수십억 개의 파라미터를 연산한다.

```
비용 = (Input Tokens × 입력 단가) + (Output Tokens × 출력 단가)
```

여기서 중요한 포인트: **출력이 입력보다 3~5배 비싸다.**

- 입력: 병렬 처리 가능. 한 번에 쭉 읽는다.
- 출력: 순차 생성. 이전 토큰이 있어야 다음 토큰을 만든다.

실제 가격을 비교하면:

| 모델 | 입력 (1M 토큰당) | 출력 (1M 토큰당) |
|------|----------|----------|
| Claude Sonnet | $3 | $15 |
| Claude Haiku | $0.80 | $4 |
| Gemini Flash | $0.075 | $0.30 |
| DeepSeek V3 | $0.27 | $1.10 |

같은 대화를 Sonnet으로 하느냐 Flash로 하느냐에 따라 **40배** 차이가 난다.

## 모델만 바꾸는 건 한계가 있다

저가 모델을 쓰면 비용은 줄지만, 품질도 떨어질 수 있다. 진짜 최적화는 **토큰 자체를 줄이는 것**이다.

### 1. Prompt Caching — 같은 말 반복하지 마

System prompt가 매 요청마다 반복된다. 500 토큰짜리 시스템 프롬프트가 100번 호출되면 50,000 토큰이 낭비된다.

Anthropic, Google 모두 prompt caching을 지원한다. 같은 프롬프트가 반복되면 캐시된 걸 쓴다. **입력 비용 90% 절감.**

### 2. 대화 압축 — 전체를 보내지 마

이게 핵심이다. 대화가 길어지면 매 요청에 전체 히스토리를 보낸다.

```
1턴:  System(500) + User(100)                    = 600 토큰
5턴:  System(500) + 이전대화(2000) + User(100)     = 2,600 토큰
10턴: System(500) + 이전대화(5000) + User(100)     = 5,600 토큰
```

10턴째에는 1턴 때의 **10배** 비용을 내고 있다.

해결 방법 3가지:

- **Sliding Window**: 최근 N개 메시지만 보냄. 간단하지만 오래된 맥락 손실.
- **요약 + 최근**: 오래된 대화를 한 줄로 요약. 5,600 → 1,100 토큰. **80% 절감.**
- **RAG**: 벡터 DB에 저장하고 관련된 것만 검색. 가장 효율적이지만 구현 복잡.

### 3. Model Routing — 어려운 것만 비싼 모델로

"안녕" 같은 간단한 인사에 Sonnet을 쓸 필요가 없다.

요청의 복잡도를 판단해서:
- 간단한 질문 → Gemini Flash (무료)
- 기술 논의 → DeepSeek 또는 Haiku (저가)
- 복잡한 설계 → Sonnet (고가)

이렇게 자동 라우팅하면 **30~50% 절감** 가능하다.

### 4. Semantic Caching — 같은 의미면 API 호출 안 함

"배달 앱 만들어줘"와 "음식 배달 서비스 개발하고 싶어"는 글자는 다르지만 의미는 같다.

사용자 입력을 벡터(embedding)로 변환해서, 이전에 비슷한 질문이 있었으면 저장된 응답을 재사용한다. API 호출 자체를 안 하니까 **비용 0.**

### 5. Output 최적화 — 짧게 답해

출력이 입력보다 3~5배 비싸니까, 응답을 짧게 만드는 것만으로도 효과가 크다.
- max_tokens 제한
- 시스템 프롬프트에 "간결하게 답변" 지시
- JSON schema로 구조화된 응답만 받기
